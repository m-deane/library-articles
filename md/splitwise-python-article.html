Splitwise in Python:

The Complete Practitioner's Guide

From OAuth handshakes to debt-minimisation graphs — everything a data engineer, automation developer, or financial-analysis practitioner needs to wire Python directly into Splitwise's REST API using the splitwise SDK.

What Is Splitwise?

Splitwise is a debt-tracking and expense-splitting platform launched in 2011, now serving tens of millions of users globally. Unlike a payment processor, it is a ledger of obligations — it records who paid what, computes who owes whom, and optimises the resulting debt graph to minimise the number of settlement transactions required.

At its core, Splitwise solves a surprisingly hard combinatorial problem: given N people with M expenses between them, find the minimum set of payments that resolves every debt. Naively, this is O(N²) bilateral settlements; Splitwise's "Simplify Debts" feature reduces this to at most N-1 transfers by treating the problem as a flow network.

The platform exposes a full REST API at https://secure.splitwise.com/api/v3.0/, supporting OAuth 1.0a, OAuth 2.0, and personal API keys. Every resource — users, groups, expenses, comments, currencies, categories — is accessible programmatically. This makes Splitwise an attractive target for financial automation, household budget analysis, and shared-economy tooling.

💡 Key Concept

Expenses not assigned to a group are placed in a virtual group with id=0. Always filter getExpenses() results by getPayment() == False to exclude settlement transactions from spending analyses.

The Python SDK Landscape

Several community SDKs exist across languages for the Splitwise API. For Python, the canonical choice is namaggarwal/splitwise, a 208-star MIT-licensed package on PyPI that provides complete typed object wrappers around every API endpoint. It is the only Python SDK listed on the official Splitwise developer documentation.

The SDK supports Python 3.7 through 3.11 and runs on PyPy. It wraps every API resource as a Python class with explicit getter and setter methods — a deliberate design choice that provides IDE autocomplete at the cost of more verbose code than a simple dict-based client.

Authentication Deep Dive

Splitwise supports three authentication mechanisms. Choosing the wrong one for your use case is the most common source of integration friction. Understanding the tradeoffs upfront saves hours of debugging.

For personal automation — monthly spending reports, household budget dashboards, CLI tools — the API key is the fastest path. Register an application at https://secure.splitwise.com/oauth_clients, then use the API key you receive directly. This method acts on your account only and is not suitable for multi-user applications.

💡 Security Note

Never commit API keys to version control. Store them in a .env file loaded via python-dotenv, or use environment variables in your CI/CD pipeline. The .env file should be in .gitignore.

For multi-user applications, OAuth 2.0 is mandatory. The SDK handles the full authorization code flow. The critical security step is validating the state parameter on callback to prevent CSRF attacks — the SDK returns it from getOAuth2AuthorizeURL() and you must compare it to what comes back from Splitwise.

"Store the access token after the first OAuth exchange — Splitwise tokens do not expire, making them safe to persist in your database for long-running automations."

Core Object Model

Every API response is deserialised into a typed Python object. Understanding the object hierarchy is essential for writing efficient, non-redundant code. The central entry point is the Splitwise class; all objects it returns are instances of the following classes from the splitwise package.

💡 Gotcha

getCost() returns a string, not a float. Always cast: float(expense.getCost()). Similarly, getOwedShare() and getPaidShare() on ExpenseUser objects are also strings.

Reading Data

The getExpenses() method is the workhorse of any Splitwise integration. It supports rich server-side filtering, but has a hard limit of 100 results per call, requiring pagination for any account with significant history.

💡 Filter Parameters

dated_after / dated_before — ISO 8601 strings

updated_after — for incremental sync

group_id — filter by group

friend_id — bilateral expenses

limit — max 100 per page

offset — pagination cursor

The pattern above — paginating until an empty page is returned, then converting to a pandas DataFrame — is the recommended foundation for any analytical pipeline. Filter getDeletedAt() immediately; deleted expenses are returned by default and will corrupt aggregations if not excluded.

For incremental pipelines, use updated_after instead of dated_after. This captures edits and deletions to previously synced expenses, which dated_after misses entirely.

Writing Data

Creating expenses programmatically requires careful attention to the paid_share and owed_share fields on each ExpenseUser. These are the most common source of API errors for new integrators: the sum of all paid_share values must equal the total expense cost, and the sum of all owed_share values must also equal the total cost.

💡 Invariant

Always verify: sum(paid_shares) == total_cost and sum(owed_shares) == total_cost. Violating either will return a validation error from the API. Float precision errors are a common cause — use round(value, 2) and adjust the last share to absorb rounding.

updateExpense() accepts a partial Expense object with only the fields to change — you do not need to re-specify the full expense. The most ergonomic pattern is to first getExpense(id), mutate the fetched object, then pass it to updateExpense(). Deletion via deleteExpense(id) is a soft delete — the expense is retained in API responses with a non-null getDeletedAt().

Analytics & Reporting

This is where the Python SDK truly differentiates itself from Splitwise's native app. By combining the SDK with pandas, you can build financial dashboards, categorised spending reports, and group fairness analyses that the app simply does not offer.

The monthly pivot table pattern is particularly useful for household financial reviews. The key insight is filtering getPayment() == True to exclude settlement transfers, which are not actual expenditures. Without this filter, a £200 Venmo repayment from a flatmate inflates your apparent spending by that amount.

Rate Limits & Best Practices

Splitwise's API does not publish formal rate limit documentation, but community experience places the practical ceiling at approximately 60 requests per minute. Scripts that iterate over large expense histories using per-expense getExpense(id) calls will hit this limit quickly — always prefer bulk operations like getExpenses() with filters over individual fetches.
