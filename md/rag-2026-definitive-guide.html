1. The Question Everyone Is Asking

It's April 2026. Your team has spent six months building a RAG pipeline — chunking 50,000 internal documents, embedding them with OpenAI's text-embedding-3-large, storing them in Pinecone, retrieving top-5 chunks per query, reranking with Cohere Rerank, and generating answers with Claude. The pipeline works. Accuracy is 87% on your evaluation set. Latency is 2.3 seconds. Cost is $0.004 per query. Then your CTO reads an article about Gemini 2.0 with its 2-million-token context window, watches a demo of an AI agent that searches the web and reads documents in real time, and asks: "Why are we maintaining all this infrastructure when the model can just read everything?"

It's a fair question. In 2023, when context windows were 8K tokens and models hallucinated constantly, RAG was the only viable architecture for knowledge-grounded AI. In 2026, four paradigm shifts — long-context models, agentic AI, fine-tuning advances, and graph-based retrieval — have fundamentally changed the calculus. RAG is not dead. But it is no longer the default answer to every knowledge-grounding problem. The decision is now: retrieval vs. context stuffing vs. fine-tuning vs. agentic tool-use — and the answer depends on scale, freshness, cost, auditability, and accuracy requirements.

This article covers everything: the original RAG formulation and why it was a breakthrough; the paradigm shifts that challenge its relevance; where RAG remains essential and where it's being replaced; the competitive landscape of frameworks and managed services; advanced patterns from Self-RAG to Agentic RAG; evaluation methodology; and an honest decision framework for 2026. Every concept section contains a runnable notebook cell — green-bordered cells 0–9 form a complete working RAG pipeline.

2. The Original RAG: Lewis et al. (NeurIPS 2020)

Retrieval-Augmented Generation was formalised by Patrick Lewis, Ethan Perez, and colleagues at Meta AI (FAIR) in their NeurIPS 2020 paper. The core idea: instead of relying solely on the parametric knowledge stored in a language model's weights, augment the generation process with non-parametric memory — a retrieval step that fetches relevant documents from an external corpus. The probabilistic formulation is: p(y|x) = Σ_z p(y|x,z) · p(z|x), where x is the input query, z is a retrieved document, and y is the generated output. The model marginalises over retrieved documents, combining retrieval probability with generation probability.

The original implementation used DPR (Dense Passage Retrieval, Karpukhin et al. 2020) as the retriever — a dual-encoder architecture that embeds queries and passages into a shared vector space, where relevance is measured by inner product. The generator was BART (Lewis et al. 2020), a sequence-to-sequence model. Two variants were proposed: RAG-Sequence (one document per generation) and RAG-Token (different documents can be retrieved for different tokens). The paper demonstrated state-of-the-art results on open-domain QA, fact verification, and knowledge-intensive NLP tasks — outperforming models with 25x more parameters.

Why this was a breakthrough: it decoupled knowledge storage from knowledge use. A language model no longer needed to memorise every fact during pre-training — it could look facts up at inference time from an updatable external store. This meant knowledge could be refreshed without retraining, attributed to specific sources, and controlled by the operator. These three properties — freshness, attribution, and control — remain the core value proposition of RAG in 2026, even as the implementation details have changed entirely.

3. Chunking: The Most Important RAG Decision

Before retrieval can work, documents must be split into chunks — segments small enough to be embedded and retrieved individually but large enough to carry meaningful context. This sounds like plumbing. In practice, chunking quality determines RAG quality more than the choice of embedding model, vector database, or LLM. A poorly chunked corpus produces irrelevant retrievals regardless of how good the rest of the pipeline is.

[Figure: Chunk Diag — see interactive JSX version]

Six chunking strategies, from simplest (fixed-size) to most sophisticated (late chunking). Each trades off simplicity against semantic coherence. Parent-document and recursive chunking are the current production defaults.

Fixed-size chunking (512 tokens with 50-token overlap) is the simplest: it's fast, deterministic, and requires no NLP. But it breaks sentences mid-thought and splits related paragraphs into separate chunks. Sentence-level chunking respects natural language boundaries but produces uneven chunk sizes. Semantic chunking uses embedding similarity to detect topic shifts and splits at natural breakpoints — more expensive but produces coherent topical chunks. Recursive chunking (LangChain's default) splits hierarchically: first by double-newlines, then by single-newlines, then by sentences — respecting document structure. Parent-document chunking retrieves small chunks for precision but returns the larger parent document for context — the best of both worlds for many use cases. Late chunking (Jina AI, 2024) embeds the full document first, then chunks after embedding — preserving cross-chunk context that other methods lose.

# Cell 1: 01_chunking.py
# Recursive chunking — the production default

    chunk_size=500,         # target chunk size in chars
    chunk_overlap=50,       # overlap to preserve context
    separators=["\n\n", "\n", ". ", " ", ""],  # hierarchy
    length_function=len,
)

chunks = splitter.split_documents(documents)
print(f"Split {len(documents)} docs into {len(chunks)} chunks")
print(f"Avg chunk size: {np.mean([len(c.page_content) for c in chunks]):.0f} chars")

4. Embeddings and Vector Search

Chunks become vectors through embedding models — neural networks that map text to dense numerical representations where semantic similarity corresponds to geometric proximity. The embedding landscape in 2026 is dominated by a handful of models: OpenAI's text-embedding-3-large (3072 dimensions, best-in-class on MTEB benchmarks), Cohere's embed-v3 (1024 dimensions, multilingual), BGE (BAAI, open-source, strong on retrieval tasks), E5-mistral (Microsoft, instruction-tuned), GTE (Alibaba), and Jina's v3 embeddings (late-interaction architecture). The choice matters less than most practitioners think — the top 5 models are within 2-3% of each other on standard benchmarks. What matters more is matching the embedding model to your domain and keeping the chunking strategy consistent.

Vector databases store these embeddings and support fast approximate nearest-neighbour (ANN) search. The landscape: Pinecone (fully managed, serverless, simplest to operate), Weaviate (hybrid search built-in, strong filtering), Qdrant (Rust-based, fast, good developer experience), Milvus (distributed, petabyte-scale), Chroma (lightweight, Python-native, great for prototyping), pgvector (PostgreSQL extension, use when you already have Postgres), FAISS (Meta, in-memory, fastest for pure vector search but no database features). For production enterprise RAG, Pinecone and Weaviate dominate; for prototyping and small-scale, Chroma; for existing Postgres infrastructure, pgvector.

# Cell 2: 02_embed_store.py
# Embed chunks and store in Chroma (local, for prototyping)
embedding_model = HuggingFaceEmbeddings(
    model_name="BAAI/bge-small-en-v1.5",  # lightweight, good quality
    model_kwargs={"device": "cpu"},

vectorstore = Chroma.from_documents(
    documents=chunks,

# For production: replace with Pinecone, Weaviate, or Qdrant

5. Retrieval and Reranking

Raw vector search returns the top-k chunks by cosine similarity — but similarity is not relevance. A chunk about "work from home policies" and a chunk about "home insurance coverage" may have similar embeddings but very different relevance to a question about remote work. This is where reranking enters: a cross-encoder model that takes the query and each candidate chunk as a pair and scores their relevance more precisely than the bi-encoder used for initial retrieval. Cross-encoders like Cohere Rerank, ColBERT (Khattab, Stanford), and BGE-reranker-v2 can improve retrieval precision by 15-30% on typical benchmarks.

Hybrid search — combining BM25 (keyword/lexical) with dense (vector) retrieval — is now the production default. BM25 catches exact term matches that dense retrieval misses (acronyms, product codes, legal clause numbers); dense retrieval catches semantic matches that BM25 misses ("What's our WFH policy?" matching a document titled "Remote Work Guidelines"). The fusion is typically done via Reciprocal Rank Fusion (RRF): score = Σ 1/(k + rank_i) across the retrieval methods. Weaviate and Elasticsearch support hybrid search natively; for other vector stores, implement it at the application layer.

The key insight: retrieval precision matters more than retrieval recall for generation quality. Retrieving 10 chunks of which 3 are relevant produces better generation than retrieving 20 chunks of which 3 are relevant — because the irrelevant chunks dilute the signal and can mislead the generator. This is the "retrieved garbage in, hallucinated garbage out" problem. Reranking is the most cost-effective intervention to fix it.

# Cell 3: 03_retrieve_rerank.py
# Basic retrieval
retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 10},  # retrieve more, then rerank
)

query = "What is the company's remote work policy?"
raw_results = retriever.invoke(query)
print(f"Retrieved {len(raw_results)} chunks")
for i, doc in enumerate(raw_results[:3]):
    print(f"  [{i}] {doc.metadata.get('topic','?')} — {doc.page_content[:80]}...")

6. The RAG Chain: Retrieval Meets Generation

The RAG chain connects retrieval to generation: retrieved chunks are formatted into a prompt, sent to an LLM alongside the user's question, and the LLM generates an answer grounded in the retrieved context. The prompt template is critical — it must instruct the model to answer only from the provided context, cite its sources, and say "I don't know" when the context doesn't contain the answer. Without these instructions, the model will happily hallucinate an answer using its parametric knowledge, defeating the purpose of retrieval.

In 2026, the RAG chain is typically orchestrated via LangChain's LCEL (LangChain Expression Language) or LlamaIndex's query engine abstraction. The key design decisions are: how many chunks to include (typically 3–7 after reranking), whether to include metadata (source document, page number, date), and how to handle multi-turn conversations (retrieve fresh context per turn, or carry forward previous context). For multi-turn, the standard approach is to reformulate the user's follow-up question into a standalone query before retrieval — otherwise the retriever sees "What about the exceptions?" without knowing which policy is being discussed.

# Cell 4: 04_rag_chain.py
from langchain_core.prompts import ChatPromptTemplate

7. The Long-Context Challenge: When Stuffing Beats Retrieval

In February 2024, Google announced Gemini 1.5 Pro with a 1-million-token context window. By late 2025, Gemini 2.0 extended this to 2 million tokens. Claude supports 200K tokens. GPT-4o handles 128K. If your entire knowledge base fits in the context window, why retrieve at all? Just stuff everything in and let the model find what it needs.

The empirical evidence is nuanced. Google's "Needle in a Haystack" tests show near-perfect recall across 1M tokens for simple factual lookups. But the "lost in the middle" phenomenon (Liu et al., 2023) persists: models attend more to the beginning and end of long contexts, with degraded attention to information in the middle. For complex reasoning over long contexts, accuracy drops significantly compared to the same information presented in a short, focused context. And then there's cost: processing 1M tokens through Gemini costs roughly 100x more than retrieving 5 relevant chunks and processing 2K tokens through a smaller model.

Long context wins when: the corpus is small (under 200K tokens total), the task is simple factual lookup, latency matters more than cost, or the question requires holistic understanding of the entire corpus (summarisation, theme extraction). RAG wins when: the corpus exceeds any context window (100K+ documents), cost matters ($0.004/query vs. $0.40/query), specific citation is required, or the knowledge changes frequently and you can't afford to re-cache the entire context. The honest answer in 2026: for most enterprise use cases, the corpus is too large and changes too frequently for context stuffing to be practical. RAG is not being replaced by long context — it's being complemented by it.

8. The Agentic AI Challenge: When the Agent Subsumes Retrieval

The more fundamental challenge to RAG comes not from longer context windows but from agentic AI architectures. An agent equipped with tool-use capabilities — web search, document reading, SQL queries, API calls, and computation — can dynamically decide what information it needs, fetch it in real time, verify it against multiple sources, and reason over it iteratively. This is fundamentally different from RAG's static pipeline, where retrieval happens once and the generator works with whatever was retrieved.

[Figure: Paradigm Shift — see interactive JSX version]

Four paradigm shifts challenging RAG's default status: long-context models, agentic AI, fine-tuning, and graph-based retrieval. Each reduces RAG's necessity for specific use cases — but none eliminates it entirely.

The Agentic RAG survey (Singh et al., arXiv 2501.09136, January 2025 — updated April 2026) provides the definitive taxonomy. Traditional RAG follows a fixed pipeline: query → retrieve → generate. Agentic RAG embeds autonomous agents into the pipeline, adding four capabilities: reflection (the agent evaluates whether the retrieved information is sufficient), planning (the agent decomposes complex queries into sub-queries), tool use (the agent can search the web, query databases, or call APIs alongside vector retrieval), and multi-agent collaboration (specialised agents handle different aspects of retrieval and generation). The key insight: in Agentic RAG, retrieval is one tool among many, not the only tool.

Anthropic's Model Context Protocol (MCP), released in late 2024, formalises this: LLMs can connect to external tools — Google Drive, Slack, databases, web search — through a standardised protocol. An agent with MCP doesn't need a pre-built RAG pipeline to access documents; it can read them directly from their source systems. This doesn't eliminate the need for vector retrieval — MCP tools are slow for searching across 100K documents — but it changes the architecture from "build a RAG pipeline" to "give the agent access to tools and let it decide when to retrieve."

Agentic RAG: the orchestrator agent decides per-query whether to retrieve from vectors, search the web, query structured data, traverse a knowledge graph, or use parametric memory — and can combine multiple sources in a single reasoning chain.

9. Self-RAG and Corrective RAG: The Model Decides When to Retrieve

Self-RAG (Asai et al., arXiv 2023) introduced a paradigm shift: instead of always retrieving, the model learns to decide when retrieval is helpful and how to use retrieved information. The model is trained with special reflection tokens that trigger retrieval, evaluate the relevance of retrieved documents, and assess whether the generated response is supported by the evidence. This addresses the core problem of naive RAG: retrieving irrelevant information is worse than not retrieving at all.

Corrective RAG (CRAG, Yan et al., arXiv 2401.15884, January 2024) takes this further. A lightweight retrieval evaluator (based on T5-large, 0.77B parameters) assesses the quality of retrieved documents and triggers three possible actions: Correct (documents are relevant → proceed with generation), Incorrect (documents are irrelevant → fall back to web search), or Ambiguous (partial relevance → combine corpus results with web search). A decompose-then-recompose algorithm selectively extracts key information from retrieved documents while filtering noise. CRAG demonstrated significant improvements over standard RAG on PopQA, Biography, PubHealth, and ARC-Challenge benchmarks.

SCMRAG (AAMAS 2025) pushed further still, integrating a dynamic knowledge graph that evolves based on the system's interactions. When information is missing, SCMRAG autonomously generates a new retrieval query and searches external sources — making the retrieval system self-correcting and adaptive. This is where RAG meets agentic AI: the system doesn't just retrieve; it reasons about what to retrieve, evaluates what it found, and corrects its own mistakes.

# Cell 5: 05_self_rag_pattern.py
# Self-RAG / CRAG pattern (conceptual implementation)
# The model decides whether retrieval is needed

def should_retrieve(query, llm_response=None):
    """Simple heuristic: retrieve if query asks about specific facts."""
    fact_indicators = ["what is", "how does", "policy", "procedure",
                       "when", "deadline", "requirement", "limit"]
    return any(ind in query.lower() for ind in fact_indicators)

def evaluate_retrieval(query, retrieved_docs, threshold=0.3):
    """CRAG-style: evaluate if retrieved docs are relevant."""
    # In production: use a trained evaluator (T5-large fine-tuned)
    # Here: simple keyword overlap heuristic
    query_words = set(query.lower().split())
    scores = []
    for doc in retrieved_docs:
        doc_words = set(doc.page_content.lower().split())
        overlap = len(query_words & doc_words) / max(len(query_words), 1)
        scores.append(overlap)
    avg_score = np.mean(scores) if scores else 0
    if avg_score > threshold:
        return "CORRECT", retrieved_docs
    elif avg_score > threshold / 2:
        return "AMBIGUOUS", retrieved_docs  # combine with web search
    else:
        return "INCORRECT", []  # fall back to web search

query = "What is the remote work policy?"
if should_retrieve(query):
    docs = retriever.invoke(query)
    action, filtered = evaluate_retrieval(query, docs)
    print(f"Query: {query}")
    print(f"Retrieval action: {action}")
    print(f"Docs passed to generator: {len(filtered)}")
else:
    print("No retrieval needed — using parametric knowledge")

10. Agentic RAG: Retrieval as One Tool Among Many

The most important architectural evolution in 2026 is the transition from RAG-as-a-pipeline to RAG-as-a-tool. In a pipeline architecture, every query triggers the same sequence: embed → retrieve → rerank → generate. In an agentic architecture, the orchestrator agent decides per-query: should I retrieve from the vector store? Search the web? Query the SQL database? Use my parametric knowledge? Combine multiple sources? The retrieval system becomes one tool the agent can invoke — not the only tool.

This is what the RAGFlow year-end review (December 2025) calls the evolution "from RAG to Context Engine" — where RAG's core capability is "intelligent retrieval" serving as the data foundation for agents. The retrieval component doesn't disappear; it becomes the knowledge layer that agents query when they need grounded, private, or structured information. The VentureBeat prediction that "contextual memory will surpass RAG for agentic AI" is accurate in one sense (agents manage their own state) but misleading in another (the memory system still needs a retrieval mechanism — it's just orchestrated differently).

In practice, Agentic RAG in 2026 looks like this: a LangGraph or LlamaIndex agent with access to multiple tools — a vector retriever for internal documents, a web search tool for live data, a SQL tool for structured data, and a calculator for numerical reasoning. The agent's planning module decomposes complex queries into sub-tasks, routes each sub-task to the appropriate tool, and synthesises the results. This is more powerful than any single RAG pipeline — but it's also more complex to build, test, evaluate, and maintain.

# Cell 6: 06_agentic_rag.py
# Agentic RAG: the agent decides which tool to use
# (Conceptual — shows the routing logic)

def route_query(query):
    """Simple router: decide which tool(s) to use per query."""
    query_lower = query.lower()

    tools_needed = []

    # Vector retrieval for internal knowledge
    if any(w in query_lower for w in ["policy","procedure","guideline","internal"]):
        tools_needed.append("vector_retrieval")

    # Web search for live/external data
    if any(w in query_lower for w in ["latest","today","current","news","price"]):
        tools_needed.append("web_search")

    # SQL for structured data
    if any(w in query_lower for w in ["how many","total","count","average","revenue"]):
        tools_needed.append("sql_query")

    # Default to vector retrieval if nothing matched
    if not tools_needed:
        tools_needed.append("vector_retrieval")

    return tools_needed

# Test routing
test_queries = [
    "What is our remote work policy?",
    "What's the latest news about AI regulation?",
    "How many employees were hired last quarter?",
    "What are the exceptions to the travel guidelines?",
]
for q in test_queries:
    tools = route_query(q)
    print(f"  '{q[:50]}...' → {tools}")

11. Where RAG Remains Essential in 2026

Despite the paradigm shifts, RAG remains the right architecture for a specific — and large — set of production use cases. Enterprise scale: organisations with 100K+ documents, petabytes of unstructured data, and thousands of users cannot stuff their corpus into any context window. Amazon's internal documentation, a hospital's medical records, a law firm's case files — these exceed any model's context capacity. RAG scales to millions of documents with sub-second latency; context stuffing does not.

Regulatory compliance and audit: In regulated industries — finance, healthcare, insurance, legal — the ability to attribute every claim in a generated response to a specific source document is non-negotiable. RAG provides provenance: "This answer was derived from Document X, Section 3, paragraph 2, last updated January 15, 2026." Pure generation cannot do this. Long-context stuffing can cite sources but with less precision. Under the EU AI Act's Article 13 transparency requirements for high-risk systems, source attribution is a compliance requirement, not a feature request.

Freshness: Knowledge that changes daily — news, pricing, inventory levels, legal filings, security advisories — cannot be captured by fine-tuning (too slow to retrain) or long-context caching (too expensive to refresh). RAG's external knowledge store can be updated in minutes. A customer support system that answers questions about this week's product releases needs retrieval, not parametric memory. Cost: For high-volume applications (thousands of queries per hour), RAG + a small model (GPT-4o-mini, Claude Haiku) costs $0.001–0.01 per query. Context-stuffing 1M tokens through a frontier model costs $0.10–1.00 per query. At enterprise scale, this is a 100x cost difference.

12. Where RAG Is Being Replaced

Simple Q&A over small document sets. If you have 50 product specification PDFs totalling 100K tokens, context stuffing with Gemini or Claude is simpler, cheaper, and more accurate than building a RAG pipeline. No chunking, no embedding, no vector database — just upload the documents and ask questions. For sub-200K-token corpora, this is now the default.

Agentic workflows where the agent can search and verify. An agent with web search, MCP document access, and multi-step reasoning can answer questions about live data more accurately than a RAG pipeline with a static index. The agent verifies its answers across multiple sources; the RAG pipeline trusts whatever it retrieves. For questions like "What did the Fed announce yesterday?" an agent with web search is categorically better than RAG over a stale corpus.

Tasks where the model already knows the answer. Asking a frontier model "What is gradient descent?" doesn't benefit from retrieval — the model's parametric knowledge is comprehensive and accurate for well-established topics. Retrieval adds 1–2 seconds of latency without improving quality. Low-stakes creative tasks. Drafting a marketing email, brainstorming product names, or writing a blog post outline — these don't need grounded retrieval. The model's creativity is the value; retrieval constraints reduce it.

14. RAG Evaluation: RAGAS and Beyond

You cannot improve what you cannot measure. RAG evaluation requires metrics for both retrieval quality and generation quality — and the interaction between them. The RAGAS framework (Shahul Es et al., 2023) defines four core metrics that have become the industry standard.

[Figure: Eval Diag — see interactive JSX version]

The four RAGAS metrics: Faithfulness (is the answer grounded in context?), Answer Relevance (does it address the question?), Context Precision (are retrieved chunks relevant?), Context Recall (did retrieval find everything needed?).

Faithfulness: Is every claim in the generated answer supported by the retrieved context? This catches hallucination — the model inventing facts not present in the retrieved chunks. Answer Relevance: Does the answer actually address the user's question? This catches off-topic responses. Context Precision: Of the retrieved chunks, how many were actually relevant? This evaluates the retriever + reranker. Context Recall: Of all the information needed to answer the question, how much was retrieved? This catches cases where relevant documents exist in the corpus but weren't found. Additional tools: TruLens (traces, feedback functions), DeepEval (unit-test-style evaluation), LangSmith (LangChain-native tracing and evaluation).

# RAG evaluation metrics (simplified implementation)
# In production: use ragas library

def faithfulness_score(answer, context):
    """Simplified: what fraction of answer words appear in context?"""
    answer_words = set(answer.lower().split())
    context_words = set(context.lower().split())
    if not answer_words:
        return 0.0
    return len(answer_words & context_words) / len(answer_words)

def context_precision(query, retrieved_docs, relevant_topics):
    """What fraction of retrieved docs are on-topic?"""
    on_topic = sum(1 for d in retrieved_docs
                   if d.metadata.get("topic") in relevant_topics)
    return on_topic / len(retrieved_docs) if retrieved_docs else 0

15. GraphRAG: When Relationships Matter More Than Similarity

Microsoft's GraphRAG (Edge et al., 2024) and the broader Graph RAG literature (surveyed in ACM Transactions on Information Systems, 2025) address a fundamental limitation of vector-based RAG: it retrieves by semantic similarity, which misses relational structure. Questions like "How does Department A's policy differ from Department B's?" or "What are the downstream effects of changing the travel reimbursement limit?" require understanding relationships between entities — not just finding similar text chunks.

GraphRAG works by constructing a knowledge graph from the corpus (entity extraction + relationship extraction via LLM), applying community detection (Leiden algorithm) to identify clusters of related entities, and generating summaries at each community level. When a query arrives, the system retrieves not just similar chunks but the relevant subgraph — entities, relationships, and community summaries. This enables global queries ("Summarise the main themes across all policy documents") that vector RAG fundamentally cannot answer because it retrieves individual chunks, not corpus-level patterns.

The tradeoff: GraphRAG is significantly more expensive to build (the indexing phase requires many LLM calls for entity/relationship extraction) and more complex to maintain (the knowledge graph must be updated when documents change). For simple factual Q&A, vector RAG is faster, cheaper, and good enough. For analytical, cross-document, or relationship-heavy queries, GraphRAG is superior. HybGRAG (ICLR 2025 submission) combines both: vector retrieval for factual queries, graph retrieval for relational queries, with a router that decides which to use per-query.

16. The Honest Verdict for 2026

[Figure: Verdict Diag — see interactive JSX version]

The 2026 verdict: RAG is essential for large-scale, fresh, cost-sensitive, citation-requiring use cases. Long context wins for small corpora. Agentic wins for multi-step reasoning. Fine-tuning wins for stable, high-volume, latency-critical domains.

RAG is not dead — but "vanilla, 2023-style RAG where you chunk documents, throw them in Pinecone, and call it a day" (to quote one industry observer) is absolutely dead. What survives and thrives is advanced RAG: hybrid search + cross-encoder reranking + query transformation + agentic orchestration + evaluation-driven iteration. The RAGFlow year-end review captures it well: RAG is evolving from "Retrieval-Augmented Generation" into a "Context Engine" — the intelligent retrieval layer that feeds agents, dashboards, and applications with the right information at the right time.

The future is probably Agentic RAG: the agent orchestrates retrieval as one tool among many, deciding per-query whether to use vector search, web search, SQL, knowledge graphs, or parametric memory. The retrieval infrastructure doesn't disappear — it becomes the data backbone that agents depend on. For practitioners, this means: invest in retrieval quality (chunking, embedding, reranking, evaluation) over framework complexity. The frameworks will change; the quality of your retrieval will determine the quality of your AI system regardless of the orchestration layer on top.

# Cell 9: 09_decision_router.py
# Decision router: which architecture for your use case?
def recommend_architecture(
    corpus_size_docs, changes_daily, needs_citations,
    queries_per_day, latency_ms_budget, multi_hop_reasoning
):
    scores = {"RAG": 0, "Long Context": 0, "Fine-Tune": 0, "Agentic": 0}

    if corpus_size_docs > 1000:    scores["RAG"] += 3
    elif corpus_size_docs < 100:   scores["Long Context"] += 3
    else:                          scores["RAG"] += 1; scores["Long Context"] += 1

    if changes_daily:              scores["RAG"] += 3; scores["Agentic"] += 2
    else:                          scores["Fine-Tune"] += 2

    if needs_citations:            scores["RAG"] += 3
    if queries_per_day > 10000:    scores["Fine-Tune"] += 2; scores["RAG"] += 1
    if latency_ms_budget < 500:    scores["Fine-Tune"] += 2; scores["Long Context"] += 1
    if multi_hop_reasoning:        scores["Agentic"] += 3

    winner = max(scores, key=scores.get)
    return winner, scores

# Example: enterprise HR chatbot
winner, scores = recommend_architecture(
    corpus_size_docs=5000, changes_daily=True, needs_citations=True,
    queries_per_day=500, latency_ms_budget=3000, multi_hop_reasoning=False
)
print(f"Recommended: {winner}")
print(f"Scores: {scores}")

19. Where RAG Goes Next

Three trajectories are clear. First, RAG becomes invisible infrastructure — embedded into model APIs (Perplexity, ChatGPT browsing, Claude web search), cloud platforms (Bedrock, Vertex, Azure), and agent frameworks (LangGraph, LlamaIndex agents). Practitioners won't "build RAG pipelines"; they'll configure retrieval as a tool in an agentic system. The RAGFlow review calls this the evolution into a "Context Engine." Second, multimodal RAG matures — retrieving not just text but images, tables, charts, and video frames. ColPali (2024) demonstrated vision-language retrieval that indexes document images directly, bypassing text extraction entirely. By 2027, RAG pipelines that handle only text will be considered incomplete.

Third, evaluation-driven RAG becomes standard practice. The gap between "we built a RAG system" and "we built a RAG system that works reliably" is measured by evaluation. RAGAS, TruLens, DeepEval, and LangSmith are making RAG evaluation as routine as model evaluation. The next frontier: continuous evaluation in production, where RAG quality metrics are monitored alongside model performance — and retrieval quality degrades trigger automatic reindexing, rechunking, or reranking threshold adjustments. RAG in 2026 is not a project you build and forget. It's a system you operate — and the organisations that treat it as operational infrastructure, not a one-time integration, will get the most value from it.
