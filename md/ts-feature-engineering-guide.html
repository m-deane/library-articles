1. Why Feature Engineering Is the Forecast

Your LightGBM model doesn't know it's forecasting a time series. It sees rows and columns — a tabular dataset like any other. The temporal intelligence — trend, seasonality, autocorrelation, external drivers — lives entirely in the features you engineer. If you create good lag features, the model learns from recent history. If you create good rolling statistics, it captures trend and volatility. If you create good calendar features, it captures seasonality. If you create bad features, or leak future information into them, no model and no hyperparameter tuning can save you.

This is the article that the Shapash time series guide assumed you'd already read. Every SHAP explanation, every Shapash dashboard, every EBM shape function explains features, not the underlying time series. "lag_1 contributed +$18k" is only meaningful if lag_1 was correctly engineered — shifted to prevent leakage, named for human readability, and grouped with related lags for stakeholder presentation. Feature engineering quality determines both forecast accuracy and explanation quality.

This article covers every feature family systematically: lags, rolling statistics, calendar, Fourier, differencing, exogenous regressors, and interactions. It covers the leakage traps that invalidate pipelines. It covers automated extraction with tsfresh and featuretools. And it covers the connection to explainability — why your feature choices determine what SHAP and Shapash can tell you. Every concept section contains a runnable notebook cell.

# Cell 0: 00_setup.py
# pip install pandas numpy scikit-learn lightgbm matplotlib
import pandas as pd
import numpy as np
from datetime import datetime, timedelta

# Generate synthetic daily demand data (2 years) with known patterns
np.random.seed(42)
n_days = 730
dates = [datetime(2022, 1, 1) + timedelta(days=i) for i in range(n_days)]
day_idx = np.arange(n_days)

trend = np.linspace(180, 220, n_days)                          # linear trend
annual = 15 * np.sin(2 * np.pi * day_idx / 365.25)             # annual cycle
weekly = 8 * np.sin(2 * np.pi * day_idx / 7)                   # weekly cycle
noise = np.random.normal(0, 5, n_days)                          # random noise
temp = 15 + 10 * np.sin(2*np.pi*(day_idx-80)/365.25) + np.random.normal(0, 3, n_days)

df = pd.DataFrame({{
    "date": dates, "value": trend + annual + weekly + noise, "temperature": temp
}})
df["date"] = pd.to_datetime(df["date"])
print(f"Generated {{len(df)}} rows: {{df['date'].min().date()}} → {{df['date'].max().date()}}")
print(f"Value range: {{df['value'].min():.0f}} – {{df['value'].max():.0f}}")

2. The Core Transformation

The fundamental idea: each row in the tabular dataset represents one forecast origin — the point in time from which you make a prediction. Each column represents one engineered feature that captures temporal information available at that origin. The target column is the value you're predicting (demand at time t+h, where h is the forecast horizon). This framing converts any time series problem into a supervised regression problem that any tabular model can solve.

[Figure: Transform Diag — see interactive JSX version]

Raw time series (left) becomes a tabular dataset (right) where each row is a forecast origin and each column is an engineered feature. The model sees only rows and columns — the temporal intelligence lives entirely in the features.

The forecast origin concept is critical for avoiding leakage. At each origin time t, you can only use information available up to time t (for a next-step forecast h=1) or up to time t−h+1 (for a direct multi-step forecast of horizon h). Any feature that incorporates information from time t+1 or later is leaking future data. This constraint determines which lags, rolling windows, and exogenous variables you can use.

For direct multi-step forecasting (predicting h steps ahead in one shot), the minimum usable lag is h itself. You cannot use lag_1 for a 7-day-ahead forecast because yesterday's value won't be known when you make the forecast 7 days in advance. This is the most common mistake in multi-step forecasting pipelines.

[Figure: Horizon Diag — see interactive JSX version]

The forecast horizon constrains which lags are available. For h=1, all lags work. For h=7, only lag_7 and beyond are valid — lag_1 through lag_6 are not yet observed at prediction time.

3. Lag Features: Carrying History

Lag features are the most important feature family for time series ML. A lag feature lag_k is simply the target value k time steps ago: lag_k(t) = y(t−k). Lag_1 carries yesterday's demand, lag_7 carries same-day-last-week, lag_365 carries same-day-last-year. Together, lags encode the autocorrelation structure — the tendency of a time series to be similar to its recent past.

Which lags to include? The ACF (Autocorrelation Function) and PACF (Partial Autocorrelation Function) tell you. Significant ACF spikes at lags 1, 7, and 14 mean the series has daily autocorrelation, weekly seasonality, and biweekly patterns. The PACF shows direct dependencies (controlling for intermediate lags) — use PACF for parsimonious lag selection. In practice, a safe default for daily data: lags 1, 2, 3, 7, 14, 21, 28, 365. For hourly: add lags 24, 48, 168 (one week).

[Figure: ACFDiag — see interactive JSX version]

ACF shows total correlation; PACF shows direct dependencies. Use PACF spikes to select which lags to engineer.

# Cell 1: 01_lags.py
# Lag features — the most important feature family
lags = [1, 2, 3, 7, 14, 21, 28]
for lag in lags:
    df[f"lag_{{lag}}"] = df["value"].shift(lag)

print("Lag features created:")
print(df[["date", "value"] + [f"lag_{{l}}" for l in lags]].tail(3).to_string(index=False))

# ACF analysis to validate lag selection
from statsmodels.tsa.stattools import acf, pacf
acf_values = acf(df["value"].dropna(), nlags=30)
print(f"\nACF at key lags: lag1={{acf_values[1]:.3f}}, lag7={{acf_values[7]:.3f}}, lag14={{acf_values[14]:.3f}}")

4. Rolling Window Statistics: Trend and Volatility

Rolling statistics aggregate recent history into summary measures: the 7-day rolling mean captures short-term trend, the 28-day rolling mean captures medium-term trend, and the rolling standard deviation captures volatility. Together, they tell the model: "where has the series been recently, how fast is it changing, and how noisy is it?"'

The critical leakage rule: rolling statistics must be computed on df["value"].shift(1), not df["value"]. If you compute df["value"].rolling(7).mean(), the window at time t includes the value at time t — which is the target you're trying to predict. Shifting by 1 before rolling ensures the window only includes past values. This is the #1 bug in time series ML pipelines, and it inflates accuracy by 5-15% while producing misleading explanations.

Beyond mean and std, consider: rolling min (recent floor), rolling max (recent ceiling), rolling median (robust central tendency), rolling skew (asymmetry), and rolling quantiles (distributional shape). The coefficient of variation (rolling_std / rolling_mean) is an underrated feature that captures whether volatility is proportional to level — important for multiplicative seasonality.

# Cell 2: 02_rolling.py
# Rolling statistics — MUST shift first to prevent leakage!
shifted = df["value"].shift(1)  # critical: use PAST values only

for window in [7, 14, 28]:
    df[f"roll_{{window}}d_mean"] = shifted.rolling(window).mean()
    df[f"roll_{{window}}d_std"]  = shifted.rolling(window).std()
    df[f"roll_{{window}}d_min"]  = shifted.rolling(window).min()
    df[f"roll_{{window}}d_max"]  = shifted.rolling(window).max()

# Coefficient of variation (volatility relative to level)
df["cv_7d"] = df["roll_7d_std"] / df["roll_7d_mean"].replace(0, np.nan)

print(f"Rolling features: {{sum(1 for c in df.columns if 'roll' in c or 'cv' in c)}} columns")
print(df[["date","roll_7d_mean","roll_7d_std","cv_7d"]].tail(3).to_string(index=False))

⚠️ The #1 leakage trap

Always compute rolling stats on df[target].shift(1), not df[target]. Without the shift, the rolling window at time t includes the value at time t — the target you're predicting. This inflates accuracy by 5-15% and produces misleading SHAP explanations. Every rolling, expanding, and EWM feature must be shifted.

5. Calendar Features: Encoding Seasonality

Calendar features capture deterministic seasonality — patterns that repeat on a known schedule. Day of week (0=Monday, 6=Sunday) captures weekly cycles (weekend vs. weekday). Month (1–12) captures annual seasonality. Hour of day captures intraday patterns. Holiday flags capture irregular but predictable events.

Encoding matters. Raw integers (month=1,2,...,12) imply an ordering that may not exist — is December "close to" January? For tree models, raw integers work because trees can split anywhere. For linear models or neural networks, cyclical encoding (sin/cos pairs) preserves the circular nature: sin(2π·month/12) and cos(2π·month/12) make December adjacent to January. One-hot encoding is safe but creates many columns for high-cardinality features (52 weeks, 365 days).

Holiday detection uses the holidays library (country-specific calendars) or workalendar (global coverage). Beyond binary holiday flags, consider: days_until_next_holiday, days_since_last_holiday, is_bridge_day (weekday between a holiday and a weekend), and holiday_type (public, school, religious) if the data supports it.

6. Fourier Features: Smooth Periodicity

Fourier features model periodic patterns as sums of sine and cosine waves at different frequencies. For a period P and harmonic k: sin(2π·k·t/P) and cos(2π·k·t/P). Each sin/cos pair captures one frequency component of the seasonality. More harmonics capture sharper patterns; fewer harmonics capture smoother patterns.

Fourier features have two advantages over raw calendar features. First, they're continuous — no discontinuity between December 31 and January 1. Second, they can model smooth seasonality with fewer features than one-hot encoding (6 Fourier pairs vs. 12 month dummies). The tradeoff: Fourier features assume smooth, repeating patterns and can't capture sharp holiday effects. Use both: Fourier for smooth annual/weekly seasonality, plus explicit holiday flags for irregular events.

How many harmonics? For annual seasonality with daily data: k=1,2,3 (3 pairs = 6 features) is usually sufficient. k=1 captures a single peak/trough; k=2 adds a second harmonic for asymmetric patterns; k=3 adds fine detail. More than 5 harmonics risks overfitting. Prophet uses this approach with built-in Fourier order selection.

# Cell 3: 03_calendar_fourier.py
# Calendar features
df["month"]       = df["date"].dt.month
df["day_of_week"] = df["date"].dt.dayofweek
df["day_of_year"] = df["date"].dt.dayofyear
df["is_weekend"]  = (df["date"].dt.dayofweek >= 5).astype(int)
df["quarter"]     = df["date"].dt.quarter
df["week_of_year"]= df["date"].dt.isocalendar().week.astype(int)

# Fourier features for smooth seasonality
for period in [7, 365.25]:
    for k in range(1, 4):  # 3 harmonics per period
        df[f"sin_{{period}}_{k}"] = np.sin(2*np.pi*k*df["day_of_year"]/period)
        df[f"cos_{{period}}_{k}"] = np.cos(2*np.pi*k*df["day_of_year"]/period)

fourier_cols = [c for c in df.columns if "sin_" in c or "cos_" in c]
cal_cols = ["month","day_of_week","day_of_year","is_weekend","quarter","week_of_year"]
print(f"Calendar: {{len(cal_cols)}} features")
print(f"Fourier: {{len(fourier_cols)}} features (3 harmonics × 2 periods × 2 trig)")

7. Exogenous Regressors

Exogenous features are external variables that influence the target but aren't derived from it — temperature for energy demand, promotion flags for retail sales, economic indicators for financial series. The key question for each exogenous variable: is it known in advance at prediction time?

Known-in-advance variables (scheduled promotions, holidays, calendar events) can be used at their concurrent value — you know next Tuesday is a holiday when forecasting next Tuesday's demand. Unknown-at-prediction-time variables (temperature, competitor pricing) must be lagged — you don't know next Tuesday's temperature, so you use today's or yesterday's temperature as a feature. Using concurrent values of unknown-at-prediction-time variables is future leakage.

# Cell 4: 04_exogenous_diff.py
# Exogenous features — lag unknown-at-prediction-time variables
df["temp_lag_1"] = df["temperature"].shift(1)
df["temp_lag_3"] = df["temperature"].shift(3)
df["temp_roll_7d"] = df["temperature"].shift(1).rolling(7).mean()

# Differencing features — capture change rates
df["diff_1"] = df["value"].diff(1)    # daily change
df["diff_7"] = df["value"].diff(7)    # weekly change
df["pct_change_1"] = df["value"].pct_change(1)  # percentage daily change
df["pct_change_7"] = df["value"].pct_change(7)  # percentage weekly change

print(f"Exogenous features: temp_lag_1, temp_lag_3, temp_roll_7d")
print(f"Differencing features: diff_1, diff_7, pct_change_1, pct_change_7")

8. Differencing: Stationarity Transforms

First differences (value(t) − value(t−1)) remove linear trend. Seasonal differences (value(t) − value(t−7)) remove weekly seasonality. Log returns (log(value(t)/value(t−1))) normalise for level-dependent volatility. These transformations make non-stationary series more amenable to tabular models — though tree-based models handle non-stationarity better than linear models, differencing features still improve accuracy by making the signal more accessible.

A practical question: should you difference the target (predict the change, then cumulate) or add differencing as features (predict the level, informed by change rates)? Both work. Differencing the target can improve stationarity but requires careful inverse-transformation at prediction time. Adding diff features is simpler and works well with tree models that can naturally handle level + change information together.

9. Interaction Features

Interaction features capture conditional effects — where one feature's influence depends on another's value. lag_1 × is_weekend captures "yesterday's demand matters differently on weekdays vs. weekends." temperature × month captures "temperature's effect on demand depends on season" (cold weather increases heating demand in winter but not in summer).

Tree models discover interactions automatically via multi-way splits. But explicitly engineering key interactions can improve convergence speed and accuracy — the model doesn't need to discover the pattern through splits, it's handed the combined signal directly. For linear models and EBMs (which are additive), explicit interactions are essential — without them, conditional effects are invisible.

# Cell 5: 05_interactions.py
# Interaction features — conditional effects
df["lag1_x_weekend"] = df["lag_1"] * df["is_weekend"]
df["lag1_x_month"] = df["lag_1"] * df["month"]
df["temp_x_month"] = df["temp_lag_1"] * df["month"]

# Clean up: drop rows with NaN from shifting/rolling
df_clean = df.dropna().reset_index(drop=True)
feature_cols = [c for c in df_clean.columns if c not in ["date", "value"]]
print(f"\nTotal features: {{len(feature_cols)}}")
print(f"Clean rows: {{len(df_clean)}} (dropped {{len(df) - len(df_clean)}} NaN rows)")
print(f"\nFeature families:")
for family, prefix in [("Lags","lag_"),("Rolling","roll_"),("Calendar","month,day,is_,quarter,week"),
                        ("Fourier","sin_,cos_"),("Exogenous","temp_"),("Diff","diff_,pct_"),("Interaction","_x_")]:
    cols = [c for c in feature_cols if any(p in c for p in prefix.split(","))]
    print(f"  {{family:15s}} {{len(cols)}} features")

10. The Leakage Problem

[Figure: Leakage Diag — see interactive JSX version]

Three leakage types. Target leakage (forgot to shift), future leakage (used concurrent exogenous), and split leakage (random train/test split). Each invalidates the pipeline.

Target leakage is the most common: computing df["value"].rolling(7).mean() includes the value at time t in the rolling window for time t's prediction. The fix: always shift before rolling. Future leakage through exogenous variables: using today's temperature to predict today's demand, when in production you won't have today's temperature at forecast time. The fix: lag all unknown-at-prediction-time exogenous variables. Split leakage: random train/test split (shuffle=True) mixes past and future, letting the model "peek" at future patterns during training. The fix: always use temporal splits (shuffle=False, or explicit date-based cutoffs).

How to detect leakage post-hoc: (1) Implausibly high accuracy — if your demand forecast achieves R²=0.99, something is wrong. (2) Permutation Feature Importance showing a feature as dramatically more important than domain knowledge suggests. (3) A large gap between train accuracy and cross-validated accuracy. (4) Walk-forward validation showing much lower accuracy than hold-out validation — the hold-out may have leaked.

# Cell 8: 08_leakage_audit.py
# Leakage audit: check for suspicious features
from sklearn.ensemble import RandomForestRegressor
from sklearn.inspection import permutation_importance

X = df_clean[feature_cols]
y = df_clean["value"]
split = int(len(df_clean) * 0.8)
X_tr, X_te = X.iloc[:split], X.iloc[split:]
y_tr, y_te = y.iloc[:split], y.iloc[split:]

rf = RandomForestRegressor(n_estimators=100, random_state=42)
rf.fit(X_tr, y_tr)
r2 = rf.score(X_te, y_te)
print(f"R² on test set: {{r2:.4f}}")

# PFI to check for suspicious features
pfi = permutation_importance(rf, X_te, y_te, n_repeats=10, random_state=42)
top5 = np.argsort(pfi.importances_mean)[-5:][::-1]
print("\nTop 5 features by PFI:")
for i in top5:
    name = feature_cols[i]
    imp = pfi.importances_mean[i]
    print(f"  {{name:25s}} {{imp:.4f}}")
print("\n⚠️ If a feature is suspiciously dominant, investigate for leakage")

11. Training the Model

With features engineered, the training step is standard tabular ML. The critical requirement: temporal train/test split. Never shuffle time series data. Use the first 80% chronologically for training and the last 20% for testing. For hyperparameter tuning, use walk-forward cross-validation (TimeSeriesSplit in scikit-learn) — not random k-fold.

LightGBM, XGBoost, and CatBoost are the typical model choices. LightGBM is fastest; CatBoost handles categoricals natively; XGBoost has the largest community. EBMs provide built-in interpretability without needing SHAP. The features we've engineered work with all of these — the tabular format is model-agnostic.

# Cell 6: 06_train.py
from lightgbm import LGBMRegressor
import lightgbm as lgb

model = LGBMRegressor(
    n_estimators=800, learning_rate=0.03, max_depth=7,
    num_leaves=64, subsample=0.8, colsample_bytree=0.8,
    reg_alpha=0.1, reg_lambda=0.1, random_state=42,
)
model.fit(X_tr, y_tr, eval_set=[(X_te, y_te)],
          callbacks=[lgb.early_stopping(50, verbose=False)])

from sklearn.metrics import mean_absolute_error, mean_absolute_percentage_error
preds = model.predict(X_te)
mae = mean_absolute_error(y_te, preds)
mape = mean_absolute_percentage_error(y_te, preds)
print(f"MAE: {{mae:.2f}}, MAPE: {{mape:.2%}}")
print(f"Best iteration: {{model.best_iteration_}}")

12. Feature Selection with PFI

After engineering 40+ features, some will be redundant or noisy. Permutation Feature Importance (PFI) on the test set identifies features that don't contribute to generalisation — remove them to simplify the model and potentially improve accuracy. The workflow: compute PFI, remove features with importance indistinguishable from zero, retrain, re-evaluate.

For time series features specifically, watch for: (1) correlated lags splitting importance (lag_1 and lag_2 each show low PFI because they carry similar information — but the pair is critical). (2) Fourier features showing zero importance on a dataset without smooth seasonality — remove them. (3) Interaction features showing zero importance — the tree model was already capturing the interaction via splits.

# Cell 7: 07_feature_selection.py
# PFI-based feature selection
threshold = 0.001
important_mask = pfi.importances_mean > threshold
important_features = [f for f, m in zip(feature_cols, important_mask) if m]
removed = [f for f, m in zip(feature_cols, important_mask) if not m]

print(f"Important: {{len(important_features)}}, Removed: {{len(removed)}}")
if removed:
    print(f"Removed features: {{', '.join(removed[:5])}}" +
          (f"... +{{len(removed)-5}} more" if len(removed) > 5 else ""))

# Retrain on selected features
model_slim = LGBMRegressor(n_estimators=800, learning_rate=0.03, max_depth=7,
    num_leaves=64, random_state=42)
model_slim.fit(X_tr[important_features], y_tr)
preds_slim = model_slim.predict(X_te[important_features])
mae_slim = mean_absolute_error(y_te, preds_slim)
print(f"\nOriginal MAE ({len(feature_cols)} features): {{mae:.2f}}")
print(f"Slim MAE     ({len(important_features)} features): {{mae_slim:.2f}}")

13. The Explainability Connection

[Figure: Explain Connect — see interactive JSX version]

Every XAI tool explains features, not the underlying time series. Feature quality and naming determine explanation quality.

SHAP says "lag_1 contributed +$18,200 to this forecast." Shapash translates this to "Yesterday's Value: $210k → +$18,200." An EBM's shape function shows exactly how f(lag_1) maps yesterday's value to today's prediction contribution. All three are explaining features. If the feature is poorly named (feature_23 instead of "Yesterday's Value"), the explanation is useless. If the feature leaks (unshifted rolling mean), the explanation is wrong. If important features are ungrouped (43 individual SHAP bars), the explanation is overwhelming.

The features_dict pattern from Shapash: map every engineered column to a human-readable label. The features_groups pattern: collapse related features into business-meaningful groups ("Recent History," "Trend," "Seasonality," "Weather"). These are not post-hoc cosmetics — they're design decisions that determine whether the explanation is actionable for non-technical stakeholders.

14. Automated Feature Engineering

[Figure: Auto FEDiag — see interactive JSX version]

Four automated FE libraries. tsfresh: comprehensive, 750+ features. featuretools: relational data. tsfel: lightweight. MLforecast: Nixtla ecosystem.

tsfresh (Christ et al., Neurocomputing 2018) automatically extracts 750+ statistical features — mean, variance, autocorrelation at multiple lags, spectral entropy, number of peaks, benford correlation, and hundreds more. It then applies hypothesis testing (Benjamini-Hochberg correction) to filter irrelevant features. Strength: discovery of unexpected patterns. Weakness: massive feature explosion (thousands of features before filtering), computational cost, and loss of interpretability — "agg_linear_trend_attr_stderr_chunk_len_50" is not a feature your VP can understand.

featuretools uses Deep Feature Synthesis to automatically generate features from relational datasets (e.g., transactions × customers × products). Better for transactional/relational time series than pure time series. tsfel is a lightweight alternative to tsfresh with fewer but well-curated features. MLforecast (Nixtla) provides lag transforms and rolling features integrated with the Nixtla forecasting ecosystem.

After automated extraction, feature selection is mandatory. Use PFI, Boruta, or RFECV to reduce tsfresh's 750+ features to a manageable set. The recommended workflow: (1) automated extraction for discovery, (2) PFI-based selection to keep only useful features, (3) manual review to verify interpretability and name features for explanation.

15. Use Case: Retail Demand Forecasting

Daily/weekly, cost-optimisation. A retail chain forecasts per-store daily demand using 35 features: lags (1, 2, 3, 7, 14, 28), rolling stats (7d, 14d, 28d mean and std), calendar (dow, month, is_weekend, is_holiday), Fourier (weekly + annual, 3 harmonics each), exogenous (promotion flag, temperature lag_1), and interactions (lag_1 × is_weekend, promotion × month). LightGBM achieves MAPE 3.2%.

The explainability payoff: Shapash shows the regional manager: "Today's forecast is $182k. Main drivers: Yesterday's Sales: $175k (+$12k), Weekend: Yes (+$8k), Active Promotion (+$6k), 7-Day Average: trending up (+$4k)." The grouping collapses 35 features into 6 business-meaningful groups. Without feature engineering, there's nothing to explain.

The leakage lesson: An early version of the pipeline used unshifted rolling_7d_mean and achieved MAPE 1.1% — suspiciously good. PFI showed rolling_7d_mean as 5x more important than any other feature. Investigation revealed the rolling window included the target day. After fixing with .shift(1), MAPE rose to 3.2% — the real performance. The 1.1% was an illusion.

16. Use Case: Energy Load Forecasting

Hourly, real-time, safety-critical. A utility forecasts hourly electricity demand using 52 features: lags (1h, 2h, 3h, 24h, 168h), rolling stats (6h, 12h, 24h, 168h), calendar (hour, dow, month, holiday), Fourier (daily + weekly + annual), weather (temperature, humidity, wind — all lagged by 1–3 hours), and interactions (temp × hour, holiday × hour). Gradient boosting achieves MAE 1.2 MW.

The calendar feature insight: Hour_of_day captures the intraday load curve (peaks at 8–9am and 6–8pm). But the interaction temp × hour captures that cold weather increases evening demand more than morning demand — a conditional effect that calendar features alone miss. This interaction feature improved MAE by 3% on winter evenings specifically.

Feature engineering for production: Features must be computable in real-time with only past data available. Temperature lags require a weather data pipeline with at most 1-hour delay. Rolling stats must use the exact same shift logic as training. Any discrepancy between training and serving feature computation invalidates the model.

17. Manual vs. Automated vs. Deep Learning

[Figure: Vs Approach — see interactive JSX version]

Three approaches: manual FE + tabular ML (this article), built-in components (Prophet), and end-to-end deep learning (TFT, N-BEATS). Each has strengths.

Manual feature engineering + tabular model (this article's approach) gives full control, interpretability, and domain knowledge injection. The features are visible, nameable, and explainable. The model is fast to train and cheap to serve. It requires engineering expertise and time. Prophet / NeuralProphet builds in trend, seasonality, holidays, and regressors as model components — faster to start, less control, adequate for many forecasting tasks. Deep learning (N-BEATS, TFT, PatchTST, Chronos) learns features end-to-end from raw sequences — no manual engineering, but requires more data, more compute, and produces less explainable models.

The empirical evidence: on structured business forecasting problems (demand, energy, sales), manual FE + gradient boosting matches or beats deep learning in most benchmarks (Makridakis M5, Kaggle time series competitions). Deep learning wins when the data is abundant, the patterns are complex, and interpretability is not required. The choice depends on your constraints: if you need explanations, manual FE + tabular. If you need automation at scale across thousands of series, Prophet or deep learning.

18. Decision Framework

✓ Use manual feature engineering when:

You need interpretable forecasts (SHAP, Shapash, EBM shape functions).

[Figure: br — see interactive JSX version]

You have domain knowledge to encode (known seasonality, known external drivers).

[Figure: br — see interactive JSX version]

The forecast will be explained to non-technical stakeholders.

[Figure: br — see interactive JSX version]

You're forecasting few series with deep history (retail store, energy grid).

[Figure: br — see interactive JSX version]

Accuracy on structured business data needs to be competitive with deep learning.

[Figure: br — see interactive JSX version]

Production latency requires fast inference (tabular models serve in milliseconds).

✗ Consider alternatives when:

You have thousands of diverse series to forecast → Prophet, Chronos, or global DL models.

[Figure: br — see interactive JSX version]

You lack domain expertise for feature design → automated FE (tsfresh) or deep learning.

[Figure: br — see interactive JSX version]

The patterns are too complex for hand-crafted features → TFT, PatchTST.

[Figure: br — see interactive JSX version]

Interpretability is not required → end-to-end DL may be simpler.

[Figure: br — see interactive JSX version]

You need uncertainty quantification → probabilistic DL (DeepAR) or conformal prediction.

Feature engineering is the bridge between time series and tabular ML. It is also the bridge between time series and explainability. Every SHAP value, every Shapash dashboard, every EBM shape function explains the features you engineered. Build them well — with correct shifting, meaningful naming, and thoughtful grouping — and the entire downstream stack works. Build them poorly, and no model, no explanation tool, and no dashboard can compensate.
