1. The Interpretability Question

You've trained an XGBoost model for credit scoring. AUC 0.923 on your holdout set. Then compliance asks: "For every denied application, provide the specific features and their exact contributions that led to the denial." You reach for SHAP — compute Shapley values, build a Shapash dashboard, generate an audit report. It works, but it's two systems: the model that decides, and the explanation that rationalises the decision post-hoc. The explanation is approximate, local, and computed after the fact.

There is another path. Instead of building a black-box model and explaining it, you build a model that is interpretable by design — one whose internal structure is directly human-readable, where each feature's effect is a curve you can plot, inspect, and edit. And this model matches the black box's accuracy. This is the Explainable Boosting Machine (EBM), developed at Microsoft Research by Rich Caruana and collaborators. It is a tree-based, cyclic gradient boosting Generalised Additive Model with automatic interaction detection.

This article covers everything at encyclopaedic depth: the mathematical lineage from GAMs through GA²M to EBM; the round-robin training algorithm; shape functions and their visualisation; the accuracy debate with evidence; the InterpretML library in full; four production use cases; the competitive landscape of glass-box models; model editing; and an honest decision framework. Every concept section contains a runnable notebook cell — Cells 0–9 form a complete end-to-end example.

# Cell 0: 00_setup.py

2. Generalised Additive Models: The Mathematical Foundation

An EBM is a Generalised Additive Model (GAM). GAMs were introduced by Hastie and Tibshirani (1986) as an extension of linear models: g(E[y]) = β₀ + f₁(x₁) + f₂(x₂) + ... + fₙ(xₙ), where g is a link function (identity for regression, logit for classification) and each fᵢ is a univariate "shape function" learned from data. The additive structure is the source of interpretability: each feature contributes independently, so you can visualise fᵢ(xᵢ) as a curve — x-axis is feature value, y-axis is contribution to prediction — and inspect it directly. No post-hoc approximation needed.

[Figure: GAMDiag — see interactive JSX version]

The lineage: GAMs provide the additive framework. GA²M (Lou et al., KDD 2013) adds pairwise interactions via the FAST algorithm. EBMs add modern ML training — bagging, cyclic gradient boosting, optional differential privacy.

The limitation of pure GAMs is accuracy. Because each feature acts independently, GAMs cannot capture interactions — where feature A's effect depends on feature B's value. Lou, Caruana, Gehrke, and Hooker addressed this in their KDD 2013 paper with GA²M: g(E[y]) = β₀ + Σfᵢ(xᵢ) + Σfᵢⱼ(xᵢ,xⱼ). The pairwise terms fᵢⱼ are 2D heatmaps — still visualisable and interpretable, but capturing the most important non-linear interactions. Their FAST algorithm efficiently ranks all n(n-1)/2 possible pairs to find the top-k worth including.

The result was surprising: GA²M models "substantially close the accuracy gap between unintelligible full-complexity models such as random forests and GAMs" (Lou et al., KDD 2013). On some datasets, GA²M outperformed the best random forest. This was the empirical foundation for the EBM: add modern training techniques to the GA²M framework and match black-box accuracy while retaining full interpretability.

3. The EBM Training Algorithm: Round-Robin Cyclic Boosting

The key innovation in EBM's training is cyclic, one-feature-at-a-time gradient boosting. Unlike XGBoost, which builds each tree using all features simultaneously (creating complex multi-feature splits), EBM trains a small tree on just one feature at a time, rotating through all features in round-robin fashion across thousands of iterations. In iteration 1, it fits a small tree to x₁'s residuals, then x₂'s, then x₃'s, ..., then xₙ's. In iteration 2, it repeats. This continues for 5,000 rounds with a very low learning rate (0.01).

[Figure: Train Diag — see interactive JSX version]

Each iteration trains one small tree per feature on current residuals. Low learning rate (0.01) ensures order doesn't matter. Shape functions accumulate across thousands of iterations.

The low learning rate is critical: each tree's contribution is scaled by 0.01, so the model takes tiny steps. The order in which features are processed within a cycle has negligible effect. Each feature's shape function fᵢ is the sum of all small trees trained on that feature across all iterations — effectively a lookup table: for each binned value of xᵢ, there's a contribution to the prediction. This IS the shape function — it's the actual model, not an approximation.

After main effects are learned, EBM applies the FAST algorithm to detect top-k pairwise interactions (default: 10), trained the same way on feature pairs. Bagging (outer_bags=16) wraps the entire process, training multiple EBMs on bootstrap samples and averaging them for stability and confidence intervals on shape functions.

4. InterpretML: The Library

InterpretML is MIT-licensed, created by Samuel Jenkins, Harsha Nori, Paul Koch, and Rich Caruana. It provides ExplainableBoostingClassifier and ExplainableBoostingRegressor with a scikit-learn-compatible API. EBMs handle mixed feature types natively — continuous features are automatically binned (256 bins), categoricals are handled directly without one-hot encoding, ordinals preserve ordering. You pass raw DataFrames with string columns and the model handles everything.

The library includes an interactive visualisation system (show()) that renders global shape function plots, local waterfall explanations, and interaction heatmaps in Jupyter. For programmatic access, ebm.term_names_, ebm.term_scores_, and ebm.term_features_ expose the full model internals — every shape function, every interaction, every contribution value.

InterpretML also includes post-hoc explanation tools (SHAP, LIME, PDP) under the same API, making it a unified framework for comparing glass-box and black-box explainability. But the EBM is the library's unique contribution — no other framework offers a model this accurate that explains itself by construction.

# Cell 1: 01_train_ebm.py
ebm = ExplainableBoostingClassifier(
    max_rounds=5000, learning_rate=0.01, max_leaves=3,
    min_samples_leaf=2, interactions=10, outer_bags=16,
    max_bins=256, random_state=42,
)
ebm.fit(X_train, y_train)
auc = roc_auc_score(y_test, ebm.predict_proba(X_test)[:, 1])
print(f"EBM AUC: {{auc:.4f}}")
# Typical: ~0.927 on Adult Income (XGBoost: ~0.930)

5. Shape Functions: The Core Interpretability Mechanism

Each shape function fᵢ(xᵢ) is a learned mapping from feature values to prediction contributions. For continuous features, it's a step function over bins. For categoricals, it's a bar chart. These are the actual model internals, not post-hoc approximations. When you view a shape plot, you see exactly how the model uses that feature — globally, for every prediction, with zero approximation error.

[Figure: Shape Diag — see interactive JSX version]

Two global shape functions from an EBM. Left: f(Age) is non-linear — peaks around 50. Right: f(Income) is monotonically increasing and saturating. These are exact learned effects, not SHAP approximations.

This is fundamentally different from SHAP. A SHAP waterfall shows attribution for a single prediction — local and approximate (for non-tree models). An EBM shape function shows the global effect across all predictions — exact and complete. You see at a glance: "Income above $80k always increases the prediction, saturating above $150k." For local explanations, EBM reads the contribution from each shape function at the instance's feature values and sums — exact, instantaneous, no Shapley sampling.

The interactive dashboard (show(ebm.explain_global())) renders all shape functions with confidence intervals. Domain experts can inspect each curve: does f(Age) make clinical sense? Does f(Credit Score) match regulatory expectations? This direct inspection is impossible with black-box models — you can only inspect SHAP approximations, which carry sampling noise and may disagree across methods.

# Cell 2: 02_global_explain.py
global_exp = ebm.explain_global()
# show(global_exp)  # interactive dashboard in Jupyter

# Programmatic access to shape functions
for i in range(min(5, len(ebm.term_names_))):
    name = ebm.term_names_[i]
    scores = ebm.term_scores_[i]
    print(f"{{name:25s}} min: {{scores.min():.3f}}, max: {{scores.max():.3f}}")

print(f"\nIntercept: {{ebm.intercept_[0]:.4f}}")
print(f"Main effects: {{sum(1 for t in ebm.term_features_ if len(t)==1)}}")
print(f"Interactions:  {{sum(1 for t in ebm.term_features_ if len(t)==2)}}")

# Cell 3: 03_local_explain.py
local_exp = ebm.explain_local(X_test.iloc[:5], y_test.iloc[:5])
# show(local_exp)  # interactive waterfall in Jupyter

# Programmatic: contributions for one instance
prediction = ebm.predict_proba(X_test.iloc[[0]])[0, 1]
print(f"P(>50K): {{prediction:.3f}}")
local_data = local_exp.data(0)
names, scores = local_data["names"], local_data["scores"]
ranked = sorted(range(len(scores)), key=lambda i: abs(scores[i]), reverse=True)
print("\nTop features:")
for idx in ranked[:8]:
    print(f"  {{names[idx]:30s}} {{scores[idx]:+.4f}}")

6. Pairwise Interactions and the FAST Algorithm

Pure additive models miss interactions — where feature A's effect depends on feature B. The classic example: education's effect on income depends on occupation. A PhD in engineering has a different income effect than a PhD in poetry. EBM captures this via pairwise interaction terms fᵢⱼ(xᵢ,xⱼ), visualised as 2D heatmaps.

The FAST algorithm (Lou et al., KDD 2013) makes this tractable. With n features, there are n(n-1)/2 possible pairs. FAST ranks all pairs by residual improvement using greedy forward selection, adds the top-k (default: 10). Because only a few interactions are included, the model stays interpretable. Empirically, the top 10 capture most interaction signal; going beyond 20 rarely helps.

Each interaction heatmap is directly inspectable: colour represents the joint contribution of two feature values. A compliance officer can verify that the Education × Occupation interaction makes business sense — no post-hoc tool needed, no approximation, no sampling artefacts.

# Cell 4: 04_interactions.py
interaction_terms = [(i,t) for i,t in enumerate(ebm.term_features_) if len(t)==2]
print(f"Detected {{len(interaction_terms)}} pairwise interactions:")
for idx, features in interaction_terms:
    name = ebm.term_names_[idx]
    importance = np.abs(ebm.term_scores_[idx]).mean()
    print(f"  {{name:40s}} |contrib|: {{importance:.4f}}")

7. The Accuracy-Interpretability Debate

Cynthia Rudin's "Stop Explaining Black Box Machine Learning Models for High Stakes Decisions and Use Interpretable Models Instead" (Nature Machine Intelligence, 2019) argued that the accuracy-interpretability tradeoff is often a myth. For structured tabular data, interpretable models achieve comparable accuracy — and the small gap, if it exists, is not worth the risk of deploying an unexplainable system in healthcare, criminal justice, or finance.

[Figure: Vs Box — see interactive JSX version]

Two paths to explainable ML. Path A (EBM): interpretable by design, explanations built in, editable. Path B (XGB + SHAP): two systems to build, explain approximately. For regulated domains, Path A is strictly preferable when accuracy matches.

The empirical evidence: Lou et al. (KDD 2012, 2013) showed GA²M matching random forest accuracy. Caruana et al. (KDD 2015) applied GA²Ms to real healthcare — ICU mortality, pneumonia risk, readmission — achieving state-of-the-art accuracy while revealing bugs black boxes hid. The famous case: a neural network learned asthma decreased pneumonia mortality risk — because asthmatics were admitted to ICU and received better care (a confound). The GA²M's shape function showed this as a negative contribution, allowing clinicians to catch and correct it. Chang et al. (NeurIPS 2021) benchmarked EBMs within 1-2% AUC of gradient boosted trees on most datasets.

Where EBMs genuinely lose: very high-dimensional feature spaces (500+ features), tasks dominated by higher-order interactions (3-way+), very large datasets where cyclic training is slower than XGBoost's parallel approach, and tasks where additive + pairwise structure cannot capture the signal. These cases exist but are a minority of production tabular problems.

8. Hyperparameter Tuning

[Figure: Param Diag — see interactive JSX version]

Key EBM hyperparameters: training control, tree structure, interaction detection, and domain constraints.

max_rounds (5000): boosting iterations per cycle — higher = finer shapes. learning_rate (0.01): step size — lower = smoother. max_leaves (3): tree depth per step — 3 leaves = one split, smooth shapes. interactions (10): auto-detected pairs. outer_bags (16): bagging for stability. max_bins (256): binning resolution. For best results, the InterpretML team recommends: 100 outer bags, 5000 rounds, LR 0.01.

The most impactful hyperparameter for accuracy is interactions: increasing from 10 to 50 can noticeably improve accuracy on datasets with many interacting features, at the cost of more heatmaps to inspect. max_leaves affects smoothness: 3 produces smooth curves; increasing to 8 or 16 allows sharper discontinuities that may overfit.

# Cell 5: 05_tuning.py
ebm_tuned = ExplainableBoostingClassifier(
    max_rounds=5000, learning_rate=0.01, max_leaves=3,
    interactions=20, outer_bags=25, max_bins=256,
    min_samples_leaf=2, random_state=42,
)
ebm_tuned.fit(X_train, y_train)
auc_t = roc_auc_score(y_test, ebm_tuned.predict_proba(X_test)[:, 1])
print(f"Tuned EBM AUC: {{auc_t:.4f}}")

9. Monotonic Constraints

In many domains, relationships are known: higher credit score must decrease default risk; higher dosage must increase drug effect; more experience must increase salary. EBMs support monotonic constraints — the shape function for a constrained feature is guaranteed monotonically increasing or decreasing. A compliance officer can verify the constraint holds globally.

This is more powerful than SHAP-based verification. With SHAP, you can check that the average contribution of credit score is negative — but individual instances may violate monotonicity. With EBM monotonic constraints, the guarantee is structural — no instance can violate it, because the shape function itself is monotonic.

# Cell 6: 06_monotonic.py
constraints = [0] * X_train.shape[1]
edu_idx = list(X_train.columns).index("EducationNum")
hours_idx = list(X_train.columns).index("HoursPerWeek")
constraints[edu_idx] = 1   # increasing
constraints[hours_idx] = 1 # increasing

ebm_c = ExplainableBoostingClassifier(
    monotone_constraints=constraints, random_state=42)
ebm_c.fit(X_train, y_train)
auc_c = roc_auc_score(y_test, ebm_c.predict_proba(X_test)[:, 1])
print(f"Constrained AUC: {{auc_c:.4f}}")
print("EducationNum shape guaranteed monotonically increasing")

10. Model Editing: Fixing What the Model Got Wrong

EBMs are the only ML model that supports post-training editing of learned relationships. Because each feature's effect is a lookup table, a domain expert can directly modify the values — changing the contribution of a specific feature level, flattening a region of a shape function, or removing a learned confound entirely. This is not retraining — it's direct surgery on the model's internals.

[Figure: Edit Diag — see interactive JSX version]

The pneumonia-asthma confound: the model learned asthma decreases risk (left) because asthmatics received ICU care. A clinician edits the shape function to correct this (right). No other ML model supports this.

The canonical example is from Caruana et al. (KDD 2015): the pneumonia risk model learned that asthma was a protective factor — patients with asthma who got pneumonia had lower mortality. Why? Because hospital protocols sent asthmatic pneumonia patients straight to ICU, where they received better care. A black-box model hides this confound. An EBM reveals it as a visible negative contribution in the asthma shape function. A clinician sees it, recognises the confound, and edits the shape function to set asthma's contribution to positive or zero — making the model safe for deployment.

This editing capability extends to fairness. If a shape function shows a learned bias (e.g., a demographic feature contributing to a lending decision), you can zero it out without retraining. The model's accuracy on the remaining features is preserved because the additive structure means removing one term doesn't affect the others.

# Cell 7: 07_editing.py
# Model editing: modify a shape function post-training
# Example: flatten CapitalGain's contribution above $10k
import copy
ebm_edited = copy.deepcopy(ebm)

# Find CapitalGain term
for i, name in enumerate(ebm_edited.term_names_):
    if name == "CapitalGain":
        scores = ebm_edited.term_scores_[i]
        # Get bin edges
        bins = ebm_edited.bins_[i][0]
        # Flatten: set all contributions above a threshold to constant
        threshold_bin = len(bins) // 2
        scores[threshold_bin:] = scores[threshold_bin]
        print(f"Edited {{name}}: flattened top half of shape function")
        break

# The edited model is still valid — just re-evaluate
auc_e = roc_auc_score(y_test, ebm_edited.predict_proba(X_test)[:, 1])
print(f"Edited EBM AUC: {{auc_e:.4f}}")

11. Use Case: Healthcare Risk Prediction

Safety-critical, FDA-adjacent. A hospital builds a pneumonia mortality risk model with 38 features — demographics, vital signs, lab results, comorbidities. The model must be inspectable by clinicians before deployment. An EBM achieves AUC 0.91 on the holdout set — matching the XGBoost baseline. The shape function for temperature reveals a U-shaped curve: both hypothermia and high fever increase risk. The shape function for blood pressure shows a threshold effect at 90 mmHg systolic. Both match clinical knowledge.

The confound discovery: The asthma shape function shows a negative contribution — asthma appears to decrease risk. A pulmonologist recognises the ICU admission confound (Caruana et al., KDD 2015): asthmatics with pneumonia are flagged for ICU, receiving better care, which reduces mortality in the training data. The clinician edits the shape function to set asthma's contribution to slightly positive. The model is now safe for deployment. With a black-box model, this confound would have been invisible.

Regulatory trail: The shape function plots are included in the model validation documentation. Each feature's learned effect is reviewed by a clinical committee. The documentation satisfies FDA software-as-medical-device guidelines and EU AI Act high-risk transparency requirements. No post-hoc explanation tool needed.

12. Use Case: Credit Scoring

Regulation-driven. Under the Equal Credit Opportunity Act (ECOA), lenders must provide specific adverse action reasons for every denied application: "Your application was declined because: (1) insufficient income, (2) short credit history, (3) high debt-to-income ratio." SHAP values can generate these reasons approximately, but EBM shape functions generate them exactly — each feature's contribution is read directly from the lookup table.

An EBM trained on 22 credit features achieves AUC 0.89 — within 0.5% of XGBoost (0.893). The shape function for credit_score is monotonically decreasing (enforced via constraint). The shape function for debt_to_income shows a sharp threshold at 0.43 where risk increases dramatically. The shape function for years_of_history is approximately logarithmic — marginal benefit decreases after 7 years. All three match domain knowledge and can be directly cited in adverse action notices.

# Cell 8: 08_vs_xgboost.py
# Compare EBM vs XGBoost on the same data
try:
    from xgboost import XGBClassifier
    # XGBoost requires encoded categoricals
    X_train_enc = X_train.copy()
    X_test_enc = X_test.copy()
    for col in X_train_enc.select_dtypes(include="object").columns:
        X_train_enc[col] = X_train_enc[col].astype("category").cat.codes
        X_test_enc[col] = X_test_enc[col].astype("category").cat.codes

    xgb = XGBClassifier(n_estimators=500, max_depth=6, learning_rate=0.1,
                         random_state=42, eval_metric="logloss")
    xgb.fit(X_train_enc, y_train)
    auc_xgb = roc_auc_score(y_test, xgb.predict_proba(X_test_enc)[:, 1])
    print(f"XGBoost AUC: {{auc_xgb:.4f}}")
    print(f"EBM AUC:     {{auc:.4f}}")
    print(f"Difference:  {{abs(auc_xgb - auc):.4f}}")
except ImportError:
    print("XGBoost not installed — install with: pip install xgboost")

The deployment advantage: With XGBoost, the compliance team needs SHAP + a dashboard + a report generator + ongoing explanation validation. With EBM, the model IS the explanation. One system to build, validate, deploy, and audit. For a lending institution processing thousands of applications daily, this operational simplicity is a significant cost reduction.

13. Use Case: Insurance Underwriting

MAIF's domain. An insurer models claim frequency with policyholder demographics, vehicle attributes, driving history, and geographic risk factors. Currently using XGBoost + Shapash for post-hoc explanations. Could an EBM replace the entire stack? The EBM achieves comparable accuracy (MAE within 2% of XGBoost) and its shape functions map directly to actuarial intuition: f(driver_age) shows the well-known U-curve (young and elderly drivers have higher risk), f(vehicle_age) shows increasing risk with age, and f(annual_mileage) shows a monotonically increasing risk.

Under GDPR Article 22 and the EU AI Act, automated pricing decisions must be explainable to the policyholder. EBM shape functions provide this: "Your premium increased because: Vehicle Age: 12 years (+€45), Annual Mileage: 25,000 km (+€32), Urban Postcode: Yes (+€28)." These are exact model contributions, not SHAP approximations. The compliance team reviews global shape functions at each model refresh — verifying no protected attributes have proxy effects.

When EBM replaces Shapash + XGBoost: when the accuracy gap is negligible and regulatory explainability is the primary concern. When it doesn't: when the portfolio has complex multi-way interaction patterns (weather × geography × vehicle type × time of year) that the pairwise constraint cannot capture — in which case XGBoost + Shapash remains the better architecture.

14. Use Case: Predictive Maintenance

Physics-validatable. A manufacturing plant models equipment failure probability using sensor data: temperature, vibration, pressure, operating hours, maintenance history. The domain expert (a mechanical engineer) needs to verify that the model's learned relationships match physical reality. An EBM's shape function for temperature shows: flat below 70°C, gradual increase 70–85°C, sharp increase above 85°C. The engineer confirms: this matches the thermal degradation curve for the bearing material. If f(temperature) had shown a random or non-physical pattern, the engineer would reject the model immediately.

With XGBoost, the engineer would see a SHAP beeswarm plot showing that high temperature generally increases failure risk — but not the specific functional form of the relationship. The EBM shape function shows the exact learned curve, which can be compared point-by-point against the material science data sheet. This is interpretability that goes beyond "which features matter" to "what specific relationship did the model learn — and is it physically correct?"'

The editing use case: The vibration sensor has a known calibration drift above 15 kHz. The engineer edits f(vibration) to flatten the contribution above 15 kHz — removing the artefact without retraining. The model's accuracy on the remaining features is preserved because the additive structure means changes to one term don't cascade.

15. Competitive Landscape

[Figure: Land Diag — see interactive JSX version]

Glass-box model landscape. EBM (green, centre) leads on accuracy among interpretable models. GAMI-Net, NAM, NODE-GAM are neural GAM variants. XGBoost (red, right) is the black-box accuracy ceiling. CALMs (arXiv 2602.16503, Feb 2026) are the newest competitor.

EBM vs. linear/logistic regression: EBMs are strictly better when non-linear relationships exist (they almost always do). Regression wins only when extreme simplicity is required (coefficient-based interpretation, massive datasets, sub-millisecond training). EBM vs. XGBoost + SHAP: comparable accuracy on most tabular datasets, but EBM provides exact global + local explanations by construction. XGBoost may be 1-3% more accurate on interaction-heavy tasks. EBM vs. GAMI-Net (neural GAM, Zebin Yang et al.): GAMI-Net uses neural networks for shape functions instead of trees; theoretically more flexible but harder to train and less stable. EBM vs. NAM (Neural Additive Models, Google Brain, Agarwal et al. 2021): similar concept with neural networks; less mature, higher computational cost. EBM vs. NODE-GAM: differentiable oblivious trees; research-stage. EBM vs. RuleFit: rule-based, less accurate, but human-readable rules. EBM vs. CALMs (arXiv 2602.16503, Feb 2026): Conditionally Additive Local Models use region-specific shape functions — multiple univariate effects per feature, each active in different input regions. CALMs claim to balance GAM interpretability with GA²M accuracy.

💡 When EBM wins vs. alternatives

EBM wins: tabular data, under 500 features, need exact global explanations, regulatory compliance, model editing, mixed feature types. XGBoost wins: interaction-heavy tasks, 500+ features, training speed priority. Linear regression wins: extreme simplicity needed, massive datasets, coefficient interpretation.

16. Production Deployment

EBMs are one of the fastest models at prediction time. Because prediction involves simple lookups (bin the feature value, read the contribution from the lookup table) and a single sum, EBMs execute in microseconds per instance — often faster than tree ensembles that must traverse hundreds of decision trees. The model's memory footprint is also small: just the lookup tables and intercept.

For deployment, EBMs are serialised via pickle or joblib, loaded in a Flask/FastAPI endpoint, and serve predictions with per-instance explanations at no additional cost. Every prediction comes with a free, exact decomposition into feature contributions. No SHAP computation, no TreeSHAP overhead, no explanation latency. The shape function lookup that produces the prediction IS the explanation.

# Cell 9: 09_deploy.py
import pickle
# Save
with open("ebm_model.pkl", "wb") as f:
    pickle.dump(ebm, f)
print("Model saved: ebm_model.pkl")

# Load and predict with explanation
with open("ebm_model.pkl", "rb") as f:
    ebm_loaded = pickle.load(f)

instance = X_test.iloc[[0]]
pred = ebm_loaded.predict_proba(instance)[0, 1]
local = ebm_loaded.explain_local(instance)
data = local.data(0)
print(f"Prediction: {{pred:.3f}}")
print("Explanation (top 5 features):")
ranked = sorted(range(len(data["scores"])), key=lambda i: abs(data["scores"][i]), reverse=True)
for i in ranked[:5]:
    print(f"  {{data['names'][i]:25s}} {{data['scores'][i]:+.4f}}")
# Zero additional latency — the explanation IS the prediction

17. Advanced Topics

Differential privacy: EBMs are the only glass-box model with built-in DP-SGD support (Nori et al., 2021). By adding calibrated noise to the gradient updates during training, EBMs can provide formal privacy guarantees — critical for healthcare and financial applications where training data contains sensitive information. EBM merging: because shape functions are lookup tables, multiple EBMs trained on different data partitions can be averaged — producing a merged model without sharing raw data. This enables federated-like learning for privacy-sensitive deployments. Calibration: EBMs produce well-calibrated probabilities out of the box due to the additive structure — the predicted probabilities closely match actual event frequencies. TalkToEBM (Bordt et al., XAI4Sci AAAI-24): a natural language interface that converts EBM shape functions to text and asks LLMs to describe, summarise, and audit models.

19. Decision Framework

✓ Use EBMs when:

Tabular data with under 500 features.

[Figure: br — see interactive JSX version]

Regulatory compliance requires exact feature-level explanations (ECOA, GDPR, EU AI Act).

[Figure: br — see interactive JSX version]

Domain experts need to inspect and validate learned relationships.

[Figure: br — see interactive JSX version]

Model editing is required (fix confounds, remove bias, inject domain knowledge).

[Figure: br — see interactive JSX version]

Mixed feature types (continuous + categorical + ordinal) without preprocessing.

[Figure: br — see interactive JSX version]

Calibrated probabilities are needed out of the box.

[Figure: br — see interactive JSX version]

Deployment needs per-prediction explanations at zero additional latency.

✗ Consider alternatives when:

500+ features (EBM training scales linearly with features but cyclic approach is slow) → XGBoost.

[Figure: br — see interactive JSX version]

Higher-order interactions (3-way+) dominate the signal → XGBoost or neural networks.

[Figure: br — see interactive JSX version]

Very large datasets where training time matters → XGBoost is faster to train.

[Figure: br — see interactive JSX version]

Non-tabular data (images, text, sequences) → deep learning.

[Figure: br — see interactive JSX version]

Extreme simplicity needed (coefficient interpretation) → logistic regression.

[Figure: br — see interactive JSX version]

Unstructured prediction tasks where interpretability is not required.

EBMs represent a genuine paradigm shift: instead of building an accurate model and explaining it afterward, you build a model that is accurate and self-explaining from the start. In a regulatory environment where the EU AI Act requires that "where an interpretable model achieves comparable accuracy, it should be preferred," EBMs are not just a technical option — they are the compliance-first architecture. The question for every new tabular ML project in 2026 should not be "which black box?" but "does an EBM match the black box?" If it does, deploy the glass box.
